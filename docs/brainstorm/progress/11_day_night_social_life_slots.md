# Day/Night Social Life With Finite POI Slots

Technical companion:
- `docs/brainstorm/progress/11_day_night_social_life_slots_technical.md`

This document formalizes the idea:
- morning and midday: work-focused behavior
- evening: settlers gather at social POIs based on preferences and availability
- finite physical slots per POI, occupied on the map
- outcomes feed concrete gameplay systems

No seasonal mechanics are used.

## 1. Daily Rhythm Proposal

## 1.1 Morning and Midday

Primary behavior:
- normal work, logistics, and production
- needs interrupts still apply

## 1.2 Evening

Primary behavior:
- settlers choose one social destination if possible
- each destination has finite capacity slots
- settlers physically move to and occupy assigned slot positions

Fallback:
- if no suitable slot found, settler uses home or idle fallback behavior

## 1.3 Night

Primary behavior:
- rest and recovery
- city-level risk resolution (theft, disorder, etc.) based on city state

Night mechanics are detailed in `docs/brainstorm/12_night_order_lighting_and_risk.md`.

## 2. POI Types and Meaningful Effects

We should define a full POI catalog by charter tier, so the evening system scales with city development.

### Charter-aligned POI catalog

Settlement:
- Plaza (`civic`)
- Campfire Commons (`entertainment_light`)

Market Town:
- Tavern (`entertainment`)
- Small Shrine (`worship`)

Chartered City:
- Great Shrine / Temple Court (`worship_major`)
- Library (`culture_science`)
- Bathhouse (`hygiene_social`)

Free City and above:
- Forum Hall (`civic_major`)
- Amphitheater (`entertainment_major`)
- Academy (`culture_science_major`)

Each POI has:
- finite capacity (for example plaza `9`)
- slot coordinates relative to building footprint
- open phases (mostly evening)

### Exact effect outputs and what they mean

Use concrete city-state outputs generated per evening, then consumed by night systems.

`entertainment_points`:
- generated by Tavern, Amphitheater, Commons
- reduces same-night disorder chance in nearby residential areas
- system support needed: `NightOrderManager` consumes this value

`worship_points`:
- generated by Shrine/Temple visits
- reduces panic and severe-outcome probability when risk events trigger
- system support needed: night event resolver reads worship mitigation

`culture_points`:
- generated by Library/Academy
- contributes to knowledge/civic progression pool (can later feed tech/culture tree)
- system support needed: new `CultureProgress` accumulator

`civic_points`:
- generated by Plaza/Forum attendance
- stabilizes civic reputation loss from minor nighttime issues
- system support needed: hook in `ReputationManager` for mitigation

`hygiene_points` (optional once sanitation exists):
- generated by Bathhouse
- reduces local sanitation penalties before night resolution
- system support needed: sanitation runtime reducer

## 3. Settler Preferences and Choice Model

Each settler gets lightweight preference weights:
- `pref_entertainment`
- `pref_worship`
- `pref_culture`
- `pref_civic`

Choice utility score example:
`utility = preferenceMatch + needFit + placeQuality - travelCost - queuePenalty`

Rules:
- choose highest utility among reachable POIs with free slots
- reserve slot before movement
- release slot on leave, timeout, or interruption

## 4. What Social Choices Influence

Keep impacts concrete and local first:
- same-evening venue occupancy and crowding pressure
- same-night city mitigation pools (`entertainment_points`, `worship_points`, `civic_points`)
- sanitation relief where hygiene venues exist

Do not rely on next-day modifiers in first implementation.

## 5. Data Model Proposal

Building content extensions:
```ts
socialVenue?: {
  venueType: 'entertainment' | 'worship' | 'culture_science' | 'civic' | 'hygiene_social'
  capacity: number
  slotOffsets: Array<{ x: number; y: number }>
  openPhases: Array<'evening'>
  effectsPerVisitor: {
    entertainmentPoints?: number
    worshipPoints?: number
    culturePoints?: number
    civicPoints?: number
    hygienePoints?: number
  }
  charterTierMin?: string
}
```

Settler runtime data:
```ts
socialProfile: {
  entertainment: number
  worship: number
  cultureScience: number
  civic: number
}
```

## 6. Runtime Integration Map

Engine:
- `TimeManager`: day-phase transition hooks
- `SettlerBehaviourManager`: evening destination planning action
- `MovementManager`: slot reservation target movement
- `BuildingsManager`: POI capacity and slot occupancy source of truth
- `PopulationManager`: settler profile
- `ReputationManager`: consumes civic mitigation output
- `NightOrderManager` (new): consumes evening output pools

Frontend:
- building panel shows capacity and occupancy
- optional social heatmap overlay
- top-bar indicator for evening participation and generated point pools

## 7. Balancing Knobs

Tunable parameters:
- maximum travel distance for evening outings
- per-venue effect output per visitor
- minimum participation threshold for generating mitigation pools
- penalty strength for "no venue available" evenings

Safe defaults:
- keep mitigation effects mild
- use hard caps per pool per night
- avoid hard-failing players for temporary capacity shortages

## 8. MVP Scope

Phase 1:
- 2 POI types only: Tavern and Plaza
- finite slot occupancy and reservation
- settler preference + travel-cost selection
- produce only `entertainment_points` and `civic_points` for same-night resolution

Phase 2:
- add Shrine and Library
- add `worship_points` and `culture_points` plus connected systems
- add UI overlays and historical charts
